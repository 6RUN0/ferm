# ferm (порт на Python)

`ferm` («For Easy Rule Making», произносится «фирм») — это фронтенд к
`iptables`. Он читает правила межсетевого экрана из структурированного
высокоуровневого языка конфигурации — с переменными, функциями,
массивами, блоками и включениями файлов — и устанавливает их в работающее
ядро, вызывая `iptables(8)` / `iptables-restore`. Он также управляет
семействами `ip6tables`, `arptables` и `ebtables`.

Цель — сделать правила одновременно лёгкими в написании *и* в чтении,
чтобы администратор тратил время на проектирование хороших правил, а не на
их дословный перенос.

## Состояние проекта

Этот репозиторий находится **в процессе миграции** с оригинальной
реализации на Perl на Python:

- **`src/pyferm/`** — порт на Python. **Фаза 1 завершена**: он разбирает
  полный язык конфигурации ferm и генерирует наборы правил `iptables`
  (семейства `ip`, `ip6`, `arp`, `eb`), с быстрым
  (`iptables-restore`) и медленным (по одному правилу) путями
  выполнения, страховкой отката `--interactive` и круговым преобразованием
  save-файла через `import-ferm`. Вывод проверяется **побайтово против
  Perl-оракула**.
- **`reference/`** — оригинальная реализация на Perl, сохранённая дословно
  в качестве семантического оракула. Её собственный набор тестов
  запускается командой `make -C reference check`.

**Фаза 2 (opt-in / экспериментальная)**: нативный бэкенд `nftables` за
флагом `--nft` транслирует ту же конфигурацию в нативный набор правил nft
и применяет его атомарно через `nft -f -`. Бэкенд по умолчанию остаётся
`iptables`, поэтому существующие конфигурации и вывод не меняются, пока не
передан `--nft`. `--nft` — opt-in и экспериментальный: несёт
задокументированные смысловые отличия (сдвиг `policy DROP` в модели
собственной таблицы, `@preserve` не поддерживается) и покрыт golden +
`nft -c`, но без дифференциального теста против Perl-оракула. Дорожная
карта находится в [`docs/ROADMAP.ru.md`](docs/ROADMAP.ru.md).

### Ветки

- **`main`** — новая основная ветка (линия релизов).
- **`develop`** — активная разработка; ответвляйте работу отсюда.
- **`python-port`** — ветка процесса портирования.
- **`master`** — заморожена; сохраняется только для исторической справки.

## Требования

- Python **3.11–3.14**
- `iptables` (включая `iptables-save` / `iptables-restore`) и ядро с
  поддержкой netfilter — во время выполнения
- [`uv`](https://docs.astral.sh/uv/) — для разработки

Обязательных зависимостей времени выполнения нет. `@resolve()` использует
[`dnspython`](https://www.dnspython.org/), если он установлен (полный набор
записей, включая `NS`/`MX`); иначе откатывается на системный stub-резолвер
(`getaddrinfo`, через `/etc/nsswitch.conf`), который отдаёт только записи
`A`/`AAAA` — прочие типы дают чёткую ошибку. Поставьте extra `dns`
(`pip install pyferm[dns]`) для полного набора записей. Stub-резолвер
обращается к системным источникам вроде `/etc/hosts` и mDNS, которые dnspython
обходит, поэтому бэкенды могут расходиться, когда эти локальные источники
отличаются от авторитетного DNS.

## Установка (автономный бинарник)

Для **Linux x86_64** (glibc **2.28** или новее) публикуется
самодостаточный бинарник. Он несёт собственную среду выполнения Python и
встроенный `dnspython`, поэтому установка Python на целевом хосте не
требуется. Он **не** включает `iptables` или `nft` — они должны уже
присутствовать в системе, поскольку ferm вызывает их для установки
правил.

Скачайте релизный архив `ferm-<version>-linux-x86_64.tar.gz` и
распакуйте его с сохранением символьных ссылок:

```sh
tar xzf ferm-<version>-linux-x86_64.tar.gz
```

Это создаёт каталог `ferm.dist/`, содержащий бинарник `ferm` и рядом с
ним символьную ссылку `import-ferm`.

### Держите бинарник внутри его каталога

Бинарник `ferm` загружает встроенные разделяемые объекты из собственного
каталога (по пути времени выполнения относительно `$ORIGIN`). **Не
перемещайте и не копируйте голый бинарник `ferm` из `ferm.dist/`** —
одинокая копия больше не находит свои библиотеки и не запускается. Чтобы
запускать его из каталога в `PATH`, создайте **символьную ссылку** на
бинарник, а не копию; `$ORIGIN` по-прежнему резолвится через ссылку:

```sh
ln -s /opt/ferm/ferm.dist/ferm /usr/local/bin/ferm
```

### Распаковывайте в каталог, принадлежащий root

ferm работает с правами root. Поскольку бинарник загружает разделяемые
объекты из собственного каталога, каталог дистрибутива, доступный на
запись для остальных пользователей или группы, позволяет локальному
злоумышленнику подложить рядом с бинарником вредоносный разделяемый
объект, который затем выполнится с правами root. **Распаковывайте в
каталог, принадлежащий root и недоступный на запись другим пользователям**
(например `/opt/ferm`, права `0755`, владелец `root`), и проверьте права
*до* первого запуска от имени root:

```sh
sudo install -d -o root -g root -m 0755 /opt/ferm
sudo tar xzf ferm-<version>-linux-x86_64.tar.gz -C /opt/ferm
ls -ld /opt/ferm /opt/ferm/ferm.dist
```

### Проверка загрузки

Релиз поставляется с файлом `SHA256SUMS`. Он защищает от случайной порчи
при передаче — это **целостность, а не аутентичность.** Совпадающая
контрольная сумма **не** доказывает, что файл не был злонамеренно
подменён: злоумышленник, способный заменить архив, способен заменить и
файл контрольных сумм.

Для аутентичности проверьте аттестацию провенанса сборки через CLI
GitHub:

```sh
gh attestation verify ferm-<version>-linux-x86_64.tar.gz --repo 6RUN0/ferm
```

Аттестация защищает только тех, кто действительно выполняет проверку,
поэтому проверяйте каждую загрузку, а не доверяйте файлу слепо.

### `@resolve()` и хостовый резолвер

`@resolve()` разрешает имена через хостовый `/etc/resolv.conf`, а ferm
работает с правами root. Если корректность правила зависит от
разрешённого адреса (например, ограничение доступа к именованному хосту),
подделанный или подменённый DNS-ответ может изменить, каким адресам
доверяет установленный набор правил. Для security-значимых правил
используйте доверенный или локальный резолвер либо задавайте статические
адреса напрямую.

## Использование

```sh
# Посмотреть сгенерированные правила, не трогая ядро (безопасный способ):
uv run ferm --noexec --lines /etc/ferm/ferm.conf

# Установить набор правил в работающее ядро (нужны права root):
uv run ferm /etc/ferm/ferm.conf

# Преобразовать существующий межсетевой экран в конфигурацию ferm:
uv run import-ferm > /etc/ferm/ferm.conf
```

Будьте осторожны, чтобы не заблокировать себе доступ к удалённой машине —
почаще используйте интерактивный режим (`--interactive`, `-i`). Он
устанавливает новый набор правил, а затем откатывается к предыдущему, если
вы не подтвердите вовремя.

Man-страница `ferm(1)` (написана в `reference/doc/ferm.pod`) — это
исчерпывающий справочник по синтаксису конфигурации.

## Разработка

Проект полностью управляется через `uv` и оркеструется через `nox`:

```sh
uv sync                              # создать .venv из uv.lock
uv run nox -s lint tests typecheck   # повседневный цикл
uv run nox -s preflight              # полный обязательный гейт
```

Избранные сессии `nox`:

| Сессия          | Назначение                                                      |
| --------------- | --------------------------------------------------------------- |
| `lint`          | проверка `ruff` lint + форматирования                           |
| `tests`         | модульные тесты + golden-набор                                  |
| `typecheck`     | `mypy` + `pyright` (`verifytypes` 100%)                          |
| `golden_oracle` | golden-вывод против Perl-оракула + дифференциальный фаззинг       |
| `coverage`      | покрытие с обязательным порогом                                 |
| `matrix`        | прогон набора на Python 3.11–3.14                               |
| `fuzz`          | дифференциальный фаззинг Hypothesis против оракула              |
| `crashfuzz`     | crash-фаззинг обоих парсеров через `atheris` (opt-in)           |
| `mutation`      | мутационное тестирование `mutmut` (opt-in, ночное)              |
| `lockout`       | контейнерный anti-lockout e2e для `--interactive` (opt-in)      |

### Тестирование

Набор основан на golden-файлах («ожидаемый вывод»): каждая фикстура
сопоставляет входной `.ferm` с зафиксированным в репозитории ожидаемым
выводом, и вывод ferm сравнивается после канонизации (таблицы и цепочки
выводятся в недетерминированном порядке). Поверх этого порт непрерывно
проверяется **дифференциально против Perl-оракула** — как на корпусе
реальных конфигураций, так и на сгенерированных Hypothesis входных
данных, — поэтому расхождения отлавливаются автоматически.
