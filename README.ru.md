# ferm (порт на Python)

`ferm` («For Easy Rule Making», произносится «фирм») — это фронтенд к
`iptables`. Он читает правила межсетевого экрана из структурированного
высокоуровневого языка конфигурации — с переменными, функциями,
массивами, блоками и включениями файлов — и устанавливает их в работающее
ядро, вызывая `iptables(8)` / `iptables-restore`. Он также управляет
семействами `ip6tables`, `arptables` и `ebtables`.

Цель — сделать правила одновременно лёгкими в написании *и* в чтении,
чтобы администратор тратил время на проектирование хороших правил, а не на
их дословный перенос.

## Состояние проекта

Этот репозиторий находится **в процессе миграции** с оригинальной
реализации на Perl на Python:

- **`src/pyferm/`** — порт на Python. **Фаза 1 завершена**: он разбирает
  полный язык конфигурации ferm и генерирует наборы правил `iptables`
  (семейства `ip`, `ip6`, `arp`, `eb`), с быстрым
  (`iptables-restore`) и медленным (по одному правилу) путями
  выполнения, страховкой отката `--interactive` и круговым преобразованием
  save-файла через `import-ferm`. Вывод проверяется **побайтово против
  Perl-оракула**.
- **`reference/`** — оригинальная реализация на Perl, сохранённая дословно
  в качестве семантического оракула. Её собственный набор тестов
  запускается командой `make -C reference check`.

**Фаза 2 (opt-in / экспериментальная)**: нативный бэкенд `nftables` за
флагом `--nft` транслирует ту же конфигурацию в нативный набор правил nft
и применяет его атомарно через `nft -f -`. Бэкенд по умолчанию остаётся
`iptables`, поэтому существующие конфигурации и вывод не меняются, пока не
передан `--nft`. `--nft` — opt-in и экспериментальный: несёт
задокументированные смысловые отличия (сдвиг `policy DROP` в модели
собственной таблицы, `@preserve` не поддерживается) и покрыт golden +
`nft -c`, но без дифференциального теста против Perl-оракула. Дорожная
карта находится в [`docs/ROADMAP.ru.md`](docs/ROADMAP.ru.md).

### Ветки

- **`main`** — новая основная ветка (линия релизов).
- **`develop`** — активная разработка; ответвляйте работу отсюда.
- **`python-port`** — ветка процесса портирования.
- **`master`** — заморожена; сохраняется только для исторической справки.

## Требования

- Python **3.11–3.14**
- `iptables` (включая `iptables-save` / `iptables-restore`) и ядро с
  поддержкой netfilter — во время выполнения
- [`uv`](https://docs.astral.sh/uv/) — для разработки

Обязательных зависимостей времени выполнения нет. `@resolve()` использует
[`dnspython`](https://www.dnspython.org/), если он установлен (полный набор
записей, включая `NS`/`MX`); иначе откатывается на системный stub-резолвер
(`getaddrinfo`, через `/etc/nsswitch.conf`), который отдаёт только записи
`A`/`AAAA` — прочие типы дают чёткую ошибку. Поставьте extra `dns`
(`pip install pyferm[dns]`) для полного набора записей. Stub-резолвер
обращается к системным источникам вроде `/etc/hosts` и mDNS, которые dnspython
обходит, поэтому бэкенды могут расходиться, когда эти локальные источники
отличаются от авторитетного DNS.

## Установка

Доступны три формы поставки. Все несут одну и ту же версию, производную
от одного и того же git-тега.

### ВНИМАНИЕ: конфигурация по умолчанию блокирует весь входящий трафик

Перед включением systemd-сервиса `ferm` внимательно прочитайте этот раздел.

Стартовый файл `/etc/ferm/ferm.conf`, устанавливаемый пакетом `.deb`,
задаёт **политику DROP для цепочки `INPUT`**. По умолчанию открыты только
два узких исключения: SSH на порту 22 (по имени сервиса `ssh`) и
минимально необходимые ICMPv6-сообщения по RFC 4890 (обнаружение соседей и
т. д.). Весь остальной входящий трафик блокируется немедленно при запуске
сервиса.

**До запуска `systemctl enable --now ferm`:**

- Добавьте каждый нужный вам сервис как drop-in фрагмент в
  `/etc/ferm/ferm.d/` (например, HTTP/HTTPS, нестандартные порты
  приложений).
- Если SSH работает на порту, отличном от 22, отредактируйте `$SSH_PORT` в
  `/etc/ferm/ferm.conf` до включения сервиса. Включение с неверным портом
  SSH заблокирует доступ к удалённой машине.
- Фрагменты `/etc/ferm/ferm.d/*.conf` исполняются с правами root.
  Держите их с владельцем `root:root` и правами `0644` или строже.
  Фрагмент, доступный на запись всем, — это вектор повышения привилегий.

**Переход с Perl-пакета `ferm`:** `.deb` `pyferm` устанавливается поверх
Perl-пакета через `Provides/Conflicts/Replaces: ferm`, но **не** включает и
не запускает `ferm.service` автоматически. Если вы рассчитывали на то, что
сервис был включён в Perl-пакете, необходимо явно включить его после
установки `pyferm`. Это сделано намеренно: иначе конфигурация по умолчанию
заблокировала бы вас при первой перезагрузке.

**Сторонние пакеты с `Depends: ferm`** (средства автоматизации, менеджеры
конфигурации) получат удовлетворённую зависимость от `pyferm`, однако любой
`systemctl enable ferm`, который они могут выполнить, применит конфигурацию
DROP по умолчанию. Проверьте, что делает ваша автоматизация, прежде чем
устанавливать пакет.

**Alpine (`.apk`, OpenRC):** предупреждение о понижении уровня защиты
(breadcrumb-файл о том, что после миграции брандмауэр больше не применяется
автоматически) работает и при обычной миграции в две транзакции
(`apk del ferm`, затем `apk add pyferm`), хотя у `apk` нет хука перед
удалением. Символьная ссылка в runlevel — это состояние администратора, а не
файл пакета, поэтому `apk del ferm` её не удаляет (апорт `ferm` в Alpine не
содержит deinstall-скриптлета, который бы это делал), а `apk add pyferm`
раскладывает свой сервис до запуска post-install, так что на момент проверки
ссылка присутствует. Проверка использует `[ -L ]` (lstat), поэтому срабатывает
и для живой, и для битой (dangling) ссылки. В любом случае безопасно: `pyferm`
не добавляет свой сервис ни в один runlevel, так что даже пропущенное
предупреждение никогда не приводит к блокировке. Пакеты `.deb`/`.rpm` снимают
снимок прежнего состояния в хуке перед установкой (выполняется до удаления
легаси-пакета) по systemd, SysV и `systemctl is-enabled`.

### PyPI (pip)

```sh
pip install ferm
```

Для полного словаря типов DNS-записей в `@resolve()` (включая `NS`/`MX`)
установите extra `dns`:

```sh
pip install ferm[dns]
```

Консольные скрипты `ferm` и `import-ferm` размещаются на `PATH` через pip.

### Нативный .deb-пакет

Скачайте `pyferm_<version>_all.deb` со страницы
[GitHub Releases](https://github.com/6RUN0/ferm/releases) и установите его:

```sh
sudo apt install ./pyferm_<version>_all.deb
```

`apt install ./...` (с явным путём через `./`) автоматически разрешает
зависимости. Не используйте `dpkg -i` напрямую, если не управляете
зависимостями вручную.

Имя пакета — `pyferm`, однако он объявляет `Provides: ferm`,
`Conflicts: ferm` и `Replaces: ferm`. При установке он удаляет
существующий Perl-пакет `ferm` и удовлетворяет пакеты, зависящие от `ferm`.

При миграции с Perl-`ferm` ваш изменённый `/etc/ferm/ferm.conf` сохраняется:
интерактивный `apt` предложит оставить его (вариант по умолчанию), а для
автоматического обновления передайте `-o Dpkg::Options::=--force-confold`,
чтобы сохранить его без запроса.

После установки сервис **не** включён. Ознакомьтесь с предупреждением
выше, настройте `/etc/ferm/ferm.conf` и добавьте фрагменты в
`/etc/ferm/ferm.d/`, а затем явно включите сервис:

```sh
systemctl enable --now ferm
```

**Применение изменений — `reload`, а не `restart`:** после правки
`/etc/ferm/ferm.conf` или фрагмента выполняйте `systemctl reload ferm`.
`reload` переустанавливает набор правил атомарно (через `iptables-restore`),
без окна, в котором фаервол выключен. `restart` сперва исполняет `ExecStop`
юнита, который **сбрасывает** правила (`ferm -F`) и на короткое время
оставляет хост открытым, пока правила не применятся заново.

**Сосуществование с другими firewall-инструментами:** ferm владеет каждой
управляемой таблицей и заменяет её целиком при каждом применении. Поэтому
правила, которые другой демон (Docker, fail2ban, libvirt) пишет в управляемую
ferm таблицу, теряются при следующем `reload`, если вы явно не переносите их
ключевым словом `@preserve`. Нативный бэкенд `--nft` вместо этого управляет
единственной таблицей `table <family> ferm` и не трогает остальные — но
`@preserve` не поддерживает.

Для **Linux x86_64** (glibc **2.28** или новее) публикуется
самодостаточный бинарник. Он несёт собственную среду выполнения Python и
встроенный `dnspython`, поэтому установка Python на целевом хосте не
требуется. Он **не** включает `iptables` или `nft` — они должны уже
присутствовать в системе, поскольку ferm вызывает их для установки
правил.

Скачайте релизный архив `ferm-<version>-linux-x86_64.tar.gz` и
распакуйте его с сохранением символьных ссылок:

```sh
tar xzf ferm-<version>-linux-x86_64.tar.gz
```

Это создаёт каталог `ferm.dist/`, содержащий бинарник `ferm` и рядом с
ним символьную ссылку `import-ferm`.

### Держите бинарник внутри его каталога

Бинарник `ferm` загружает встроенные разделяемые объекты из собственного
каталога (по пути времени выполнения относительно `$ORIGIN`). **Не
перемещайте и не копируйте голый бинарник `ferm` из `ferm.dist/`** —
одинокая копия больше не находит свои библиотеки и не запускается. Чтобы
запускать его из каталога в `PATH`, создайте **символьную ссылку** на
бинарник, а не копию; `$ORIGIN` по-прежнему резолвится через ссылку:

```sh
ln -s /opt/ferm/ferm.dist/ferm /usr/local/bin/ferm
```

### Распаковывайте в каталог, принадлежащий root

ferm работает с правами root. Поскольку бинарник загружает разделяемые
объекты из собственного каталога, каталог дистрибутива, доступный на
запись для остальных пользователей или группы, позволяет локальному
злоумышленнику подложить рядом с бинарником вредоносный разделяемый
объект, который затем выполнится с правами root. **Распаковывайте в
каталог, принадлежащий root и недоступный на запись другим пользователям**
(например `/opt/ferm`, права `0755`, владелец `root`), и проверьте права
*до* первого запуска от имени root:

```sh
sudo install -d -o root -g root -m 0755 /opt/ferm
sudo tar xzf ferm-<version>-linux-x86_64.tar.gz -C /opt/ferm
ls -ld /opt/ferm /opt/ferm/ferm.dist
```

В качестве подстраховки автономный бинарник проверяет это сам: при запуске
от root он отказывается стартовать, если его каталог dist не принадлежит
root либо доступен на запись группе или посторонним, и печатает, как
исправить права. Это сеть безопасности, а не замена корректной установке —
она не способна обнаружить вредоносный разделяемый объект, уже подложенный
до проверки. Переменная `FERM_SKIP_DIST_PERM_CHECK=1` отключает её для
сознательно нестандартной раскладки.

### Проверка загрузки

Релиз поставляется с файлом `SHA256SUMS`. Он защищает от случайной порчи
при передаче — это **целостность, а не аутентичность.** Совпадающая
контрольная сумма **не** доказывает, что файл не был злонамеренно
подменён: злоумышленник, способный заменить архив, способен заменить и
файл контрольных сумм.

Для аутентичности проверьте аттестацию провенанса сборки через CLI
GitHub:

```sh
gh attestation verify ferm-<version>-linux-x86_64.tar.gz --repo 6RUN0/ferm
```

Аттестация защищает только тех, кто действительно выполняет проверку,
поэтому проверяйте каждую загрузку, а не доверяйте файлу слепо.

### `@resolve()` и хостовый резолвер

`@resolve()` разрешает имена через хостовый `/etc/resolv.conf`, а ferm
работает с правами root. Если корректность правила зависит от
разрешённого адреса (например, ограничение доступа к именованному хосту),
подделанный или подменённый DNS-ответ может изменить, каким адресам
доверяет установленный набор правил. Для security-значимых правил
используйте доверенный или локальный резолвер либо задавайте статические
адреса напрямую.

## Использование

```sh
# Посмотреть сгенерированные правила, не трогая ядро (безопасный способ):
uv run ferm --noexec --lines /etc/ferm/ferm.conf

# Установить набор правил в работающее ядро (нужны права root):
uv run ferm /etc/ferm/ferm.conf

# Преобразовать существующий межсетевой экран в конфигурацию ferm:
uv run import-ferm > /etc/ferm/ferm.conf
```

Будьте осторожны, чтобы не заблокировать себе доступ к удалённой машине —
почаще используйте интерактивный режим (`--interactive`, `-i`). Он
устанавливает новый набор правил, а затем откатывается к предыдущему, если
вы не подтвердите вовремя.

### История конфигурации и откат (etckeeper)

Когда `/etc` управляется [etckeeper](https://etckeeper.branchable.com/),
каждое успешное применение фиксирует в истории `/etc` коммит с
**семантическим** сообщением о том, что изменилось в наборе правил ядра
(например, `filter/INPUT: +3 -1`), а не просто текстовый diff `.ferm`-файлов.
Это включено по умолчанию при наличии `etckeeper`; флаг `--no-etckeeper`
отключает это для отдельного запуска.

```sh
# Показать историю ревизий конфигурации ferm (только git):
ferm rollback --list

# Отменить последнее изменение: вернуть /etc/ferm к предыдущей ревизии и
# переприменить (показывает diff и запрашивает подтверждение):
ferm rollback

# Откатиться к точной ревизии:
ferm rollback --to <sha>
```

Замечания и границы:

- **Откат работает только с git.** Сторона коммита работает с любой VCS,
  поддерживаемой etckeeper; откат требует git (для других VCS он сообщает о
  недоступности с понятным сообщением).
- **Откат возвращает исходник, затем перегенерирует.** Он возвращает
  конфигурацию `/etc/ferm` и переприменяет её — он не восстанавливает точные
  байты, которые были активны раньше (они могут отличаться после обновления
  ferm/nftables или дрейфа `@resolve()`). Для межсетевого экрана это обычно
  то, что нужно; байт-точное восстановление не гарантируется.
- **Коммит снимает весь `/etc`** (природа etckeeper), поэтому попутные
  незакоммиченные изменения `/etc` попадут в коммит, атрибутированный ferm.
- **Содержимое и сообщение — разные оси.** Коммит фиксирует *исходник*
  `/etc`; сообщение описывает *дельту ядра*. Правка `.ferm`-файла без эффекта
  на ядро даст пустое тело сообщения, а чистое дерево (перезагрузка,
  `systemctl reload` или идемпотентный повтор) фиксируется молча как
  «нечего коммитить».
- Откат отказывается выполняться, если в `/etc/ferm` есть незакоммиченные
  изменения (checkout перезатёр бы их) — сперва закоммитьте или сделайте
  stash.
- Если вы откатываетесь под `--interactive` и затем **не** подтверждаете,
  ядро возвращается к состоянию до отката, а рабочее дерево остаётся
  откатанным; повторно запустите `ferm`, чтобы синхронизировать.

Man-страница `ferm(1)` (написана в `reference/doc/ferm.pod`) — это
исчерпывающий справочник по синтаксису конфигурации.

### Использование nftables-бэкенда

Бэкенд по умолчанию управляет `iptables`/`iptables-restore`. Есть два способа
сделать так, чтобы ваши правила оказались в подсистеме ядра `nft`.

**1. Переключить системный `iptables` на nft-вариант (рекомендуется,
стабильно).** В Debian / Ubuntu сама команда `iptables` — это альтернатива
между legacy- и nft-реализацией. Выберите nft-вариант, и ferm менять не
нужно — его `iptables-restore` начнёт писать прямо в nft-таблицы ядра:

```sh
sudo update-alternatives --set iptables  /usr/sbin/iptables-nft
sudo update-alternatives --set ip6tables /usr/sbin/ip6tables-nft
# или интерактивно: sudo update-alternatives --config iptables
```

**2. Нативный бэкенд ferm `--nft` (opt-in, экспериментальный).** Он
транслирует конфигурацию в нативный nft-набор правил (см. *Состояние проекта*
выше). Установите `nftables` (это Recommends пакета deb/rpm; в apk
recommends не объявлены, поэтому в Alpine установите его явно через
`apk add nftables`) и передайте `--nft`:

```sh
ferm --noexec --lines --nft /etc/ferm/ferm.conf   # посмотреть
sudo ferm --nft /etc/ferm/ferm.conf               # применить
```

Чтобы его использовала служба **systemd**, добавьте drop-in-переопределение
через `sudo systemctl edit ferm` (пустые присваивания сбрасывают значения
юнита перед переустановкой — systemd требует этого для переопределения
`ExecStart`):

```ini
[Service]
ExecStart=
ExecStart=/usr/bin/ferm --nft /etc/ferm/ferm.conf
ExecReload=
ExecReload=/usr/bin/ferm --nft /etc/ferm/ferm.conf
ExecStop=
ExecStop=/usr/bin/ferm --nft -F /etc/ferm/ferm.conf
```

В Alpine (**OpenRC**) добавьте `--nft` к вызовам `ferm` в `start()` и
`stop()` в `/etc/init.d/ferm`; `reload()` делегирует в `start()`, поэтому
наследует флаг автоматически.

## Разработка

Проект полностью управляется через `uv` и оркеструется через `nox`:

```sh
uv sync                              # создать .venv из uv.lock
uv run nox -s lint tests typecheck   # повседневный цикл
uv run nox -s preflight              # полный обязательный гейт
```

Избранные сессии `nox`:

| Сессия          | Назначение                                                      |
| --------------- | --------------------------------------------------------------- |
| `lint`          | проверка `ruff` lint + форматирования                           |
| `tests`         | модульные тесты + golden-набор                                  |
| `typecheck`     | `mypy` + `pyright` (`verifytypes` 100%)                          |
| `golden_oracle` | golden-вывод против Perl-оракула + дифференциальный фаззинг       |
| `coverage`      | покрытие с обязательным порогом                                 |
| `matrix`        | прогон набора на Python 3.11–3.14                               |
| `fuzz`          | дифференциальный фаззинг Hypothesis против оракула              |
| `crashfuzz`     | crash-фаззинг обоих парсеров через `atheris` (opt-in)           |
| `mutation`      | мутационное тестирование `mutmut` (opt-in, ночное)              |
| `lockout`       | контейнерный anti-lockout e2e для `--interactive` (opt-in)      |

### Тестирование

Набор основан на golden-файлах («ожидаемый вывод»): каждая фикстура
сопоставляет входной `.ferm` с зафиксированным в репозитории ожидаемым
выводом, и вывод ferm сравнивается после канонизации (таблицы и цепочки
выводятся в недетерминированном порядке). Поверх этого порт непрерывно
проверяется **дифференциально против Perl-оракула** — как на корпусе
реальных конфигураций, так и на сгенерированных Hypothesis входных
данных, — поэтому расхождения отлавливаются автоматически.
