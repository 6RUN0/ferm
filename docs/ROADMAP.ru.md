# Дорожная карта ferm

Это высокоуровневая дорожная карта развития ferm от оригинальной
программы на Perl к реализации на Python с нативным бэкендом `nftables`.
Здесь приведено резюме стратегии и разбивки по фазам.

**Текущее состояние:** Фаза 1 (верный порт Perl → Python с сохранением
вывода `iptables`) завершена. Фаза 2 (нативный nft-бэкенд за флагом
`--nft`) доступна, но **opt-in / экспериментальная** — функциональна и
прошла адверсариальное ревью, но без дифференциального теста против
Perl-оракула и с задокументированными отличиями по DROP-политике и
`@preserve`. Фаза 3 (упаковка) выпустила первую альфу на PyPI (`ferm`), с
пакетами `.deb`/`.rpm`/`.apk` и автономным бинарником. Фазы 4 (операционная
безопасность: `--plan`, применение дельты, откат через etckeeper) и 5
(nft-нативные множества, interval-множества, verdict-map, нативный
`reject-with`) реализованы в ветке `python-port`, но ещё не выпущены. По
умолчанию остаётся бэкенд `iptables`. См.
[`CHANGELOG.ru.md`](../CHANGELOG.ru.md).

## Ведущий принцип: одна переменная на фазу

Каждая фаза меняет ровно одну вещь — поэтому регрессии остаются
атрибутируемыми, а Perl-оракул в `reference/` продолжает стеречь поведение:

- **Фаза 1** меняет только *язык* (Perl → Python; вывод остаётся iptables).
- **Фаза 2** меняет только *цель вывода* (iptables → нативный nft).
- **Фаза 3** меняет только *упаковку*.

## Северная звезда — инварианты, которые нельзя потерять

Именно за них выбирают ferm вместо ufw / firewalld, и они переезжают
дословно. Размытие любого из них — регрессия, а не прогресс:

- **Выразительный DSL** — переменные, функции, вложенные блоки. Это ядро
  ferm (вычислитель `enter()` со скоупами и copy-on-write); переезжает 1:1
  и стережётся golden-тестами.
- **Полнота** — практически всё, что умеет выразить iptables (~99.99%
  словаря модулей). Именно полнота отличает ferm от «простых» обёрток;
  поэтому nft-бэкенд будет транслировать в *нативные* nft-выражения, а не
  опираться на слой совместимости `iptables-nft`.
- **Декларативность + атомарное применение** — никаких императивных
  up/down-шелл-скриптов. Порт сохраняет атомарное применение
  (`execute_fast` сегодня → будущий `flush table inet ferm`).

## Фазы

### Фаза 0 — Скаффолдинг и реструктуризация ✅

Перенести Perl-код в `reference/` (история сохраняется; он остаётся
семантическим оракулом) и развернуть Python-проект в корне репозитория:
`src/pyferm/`, `tests/`, `pyproject.toml`, `noxfile.py`, `pre-commit`.
Тулчейн: `uv`, `nox`, `ruff`, `mypy`/`pyright`, `pytest`, src-layout на
`hatchling`, два console-script (`ferm`, `import-ferm`).

### Фаза 1 — Верный порт Perl → Python (вывод остаётся iptables) ✅

Портировать оба скрипта — `ferm` и `import-ferm` — с сохранением
поведения. Ключевой архитектурный шов: финальное форматирование опций
выносится из ядра в интерфейс `Backend` (`render` / `commit` /
`rollback` / `capture_previous`), так что ядро отдаёт бэкенду структурное
правило `(name, values, kind, module)` + family. Бэкенд `iptables`
форматирует его; в Фазе 2 над тем же швом встаёт адаптер `nft.py`, не
трогая ядро. Пять санкционированных отступлений от дословного порта 1:1
задокументированы, и ни одно не меняет дефолтное поведение (поэтому
golden-тесты зелёные).

### Фаза 2 — Нативный nft-бэкенд ✅

Транслировать структурное правило в **нативные nft-выражения** (не в
compat `iptables-nft`), сохраняя полный словарь модулей и атомарное
применение. Входной долг из ревью Фазы 1 (мёртвый шов `read_previous`,
eb-снапшот за бэкендом, лимит глубины парсера, политика декодирования
latin-1) погашен первым.

Принятые решения и их следствия:

- **Формат генерации: текст nft, не JSON.** Бэкенд выдаёт *текстовый*
  протокол nft и применяет его атомарно через `nft -f -`, без
  зависимости от сборки nft с JSON / `libjansson`. Текст выбран как
  *переносимый пол*: он работает на любом `nft`, что важно для
  статического бинарника Фазы 3, вызывающего тот `nft`, что есть у
  пользователя, — JSON-поддержка, хоть и почти универсальна в
  дистрибутивных сборках, остаётся build-time опцией, которой некоторые
  сборки (напр. Gentoo с выключенным USE `json`) не имеют.
  Рантайм-детектируемый JSON-провод (`nft -j -f -`) с текстовым
  fallback'ом отложен в Фазу 4/5, а не отвергнут. `--nft` — строго
  opt-in; по умолчанию остаётся `iptables`.
- **Сдвиг семантики DROP-политики в модели собственной таблицы.** Бэкенд
  владеет единственной таблицей `table <family> ferm` и не перехватывает
  монолитные ядерные цепочки `INPUT` / `FORWARD` / `OUTPUT`. Базовые
  цепочки ferm сосуществуют с базовыми цепочками других таблиц на том же
  hook (упорядоченные по приоритету), поэтому пакет может быть принят
  более приоритетной чужой цепочкой раньше, чем достигнет цепочки ferm;
  следовательно, `policy DROP` в `table ip ferm` ведёт себя иначе, чем
  монолитный `INPUT DROP` в iptables. Это задокументированное, ожидаемое
  поведение модели собственной таблицы. Администраторам, которым нужна
  точная семантика монолитного DROP, следует оставаться на бэкенде
  `iptables` по умолчанию.
- **`@preserve` не поддерживается под `--nft`** — его использование
  является чистой, явной ошибкой, намеренной регрессией opt-in-бэкенда.
  Бэкенд `iptables` по умолчанию сохраняет `@preserve` без изменений.
- **NAT с портом требует транспортного матча** — `DNAT`/`SNAT` на
  `addr:port` либо `REDIRECT`/`MASQUERADE to-ports` без предшествующего
  `proto tcp`/`udp` даёт чистую ошибку на этапе трансляции, поскольку nft
  отвергает такое отображение при применении.

Послемёрджевое адверсариальное ревью (3 цикла) закрыло оставшиеся
пробелы: операнд протокола теперь валидируется наравне с остальными
стоками (это был последний fail-open-инъекционный сток),
`--nft --interactive --shell` выдаёт реальный анти-локаут снимок/восстановление,
сбой снимка для отката прерывает выполнение вместо удаления существующей
таблицы, а golden-покрытие пополнилось кейсами негации, dual-stack и
NAT-с-портом. `--nft` остаётся opt-in/экспериментальным: нет
дифференциального теста против Perl-оракула (только golden + `nft -c`), а
сдвиг DROP-политики и регрессия `@preserve` выше — намеренные; бэкенд
`iptables` по умолчанию не изменён.

### Фаза 3 — Упаковка (опционально) ✅

Опционально поставлять самодостаточный бинарник (Nuitka). Порядко-
независима: может выйти в любой момент после Фазы 1.

Из одного источника и одной git-тег-версии (`hatch-vcs`, привязан к тегам
`py-v<PEP440>` порта) теперь выходят релизные артефакты: автономный бинарник
Nuitka, **PyPI wheel + sdist** (сборка через `uv build`/`uv publish` с
Trusted Publishing, без токенов-секретов), **нативный `.deb`**
(`debhelper`/`dh-python`, drop-in поверх Perl-`ferm` через
`Provides/Conflicts/Replaces`) и нативные пакеты **`.rpm`** и **`.apk`** для
RPM-дистрибутивов и Alpine (OpenRC). `.deb`/`.rpm`/`.apk` поставляют
стартовый `/etc/ferm/ferm.conf` и сервисный юнит, который **не** включается
на установке (анти-локаут); администратор явно включает его командой
`systemctl enable --now ferm` (`rc-update add ferm` в Alpine). Аттестация
провенанса сборки охватывает бинарник, wheel, sdist и `.deb`; `.rpm`/`.apk`
пока не аттестуются.

#### Отложенный долг по упаковке

Следующие пункты явно **выходят за рамки Фазы 3** и зафиксированы здесь,
чтобы не потерять их.

**YAGNI-откладывания** (низкий спрос или легко добавить позже):

- *One-file-бинарник* — сборка в единый файл (например, режим `--onefile`
  Nuitka) вместо распакованного каталога `*.dist/`.
- *Колесо/бинарник для aarch64* — драйвер параметризован по `--arch` лишь
  для имени артефакта, но реальная нога aarch64 — это больше, чем строка
  матрицы: драйвер сейчас жёстко падает на любой не-`x86_64`, а сборочный
  образ запинен по digest на `manylinux_2_28_x86_64`. Нужны базовый образ
  aarch64 (свой digest), `--platform`/QEMU или нативный arm-раннер и снятие
  arch-гарда.
- *musl / полностью статические сборки* — для Alpine-подобных окружений или
  полностью безгликсовых систем.
- *Slim-вариант без dnspython* — уменьшенный бинарник без опциональной
  зависимости от DNS-резолвера для пользователей, которые никогда не
  используют `@resolve()`.
- *Альтернативные упаковщики* (PyInstaller, staticx) — выбранный
  инструмент — Nuitka; альтернативы не запрещены, но не приоритетны.
- *GPG- / Sigstore-подпись ключами мейнтейнера* — аттестация
  происхождения сборки (SLSA через GitHub Actions) уже входит в Фазу 3;
  подпись в keyring — отдельный, отложенный шаг.
- *Man-страница из POD* — источник в POD существует только для версии на
  Perl; юнит `.deb` ссылается в `Documentation=man:ferm(1)` на man-страницу,
  которая для порта ещё не сгенерирована.
- *Собственный apt-репозиторий* (reprepro/aptly) — `.deb` поставляется как
  Release Asset GitHub; подписанный apt-репозиторий — отдельный шаг
  жизненного цикла дистрибуции.

**Кросс-дистрибутивное предупреждение о понижении защиты (проверено):**
breadcrumb о понижении уровня защиты (предупреждение, что после миграции
легаси-`ferm` → `pyferm` брандмауэр больше не применяется автоматически)
срабатывает во всех трёх форматах. В `.deb`/`.rpm` хук перед установкой (deb
`preinst` / rpm `%pre`) снимает снимок прежней включённости по **всем**
режимам — символьная ссылка systemd в wants, SysV-ссылки запуска
`rc[2-5].d`, а также best-effort `systemctl is-enabled` — в маркер `/run`,
который шаг post-install затем потребляет, *до* удаления легаси-пакета. В
Alpine обычная миграция выполняется в две транзакции (`apk del ferm`, затем
`apk add pyferm`), и у `apk` нет хука перед удалением, но это не проблема:
символьная ссылка в runlevel — состояние администратора, которое `apk del` не
удаляет (апорт `ferm` не содержит deinstall-скриптлета), а `apk add pyferm`
заново раскладывает свой сервис до запуска post-install, поэтому ссылка
присутствует; проверка использует `[ -L ]` (lstat), так что срабатывает и для
живой, и для битой ссылки. В любом случае безопасно: сервис не добавляется ни
в один runlevel, так что даже пропущенное предупреждение никогда не приводит к
блокировке.

**CVE-пересборка bundled-нативных библиотек (автоматизируемый триггер
РЕАЛИЗОВАН; восстановление остаётся постоянным долгом владельца):**

`pip-audit` покрывает только Python-зависимости. Он *не* сканирует
нативные разделяемые объекты, замороженные в дистрибутиве, — копии
OpenSSL, libffi, xz, bzip2 и mpdecimal, подтянутые из базового образа
сборки. OpenSSL, поставляемый базовым образом сборки `manylinux_2_28`,
относится к **серии 1.1.x, которая завершила поддержку upstream**, что
делает автоматизируемый триггер CVE-пересборки особенно важным.

Автоматизируемый триггер теперь на месте (`packaging/scan_image.py`,
запуск `uv run nox -s image_scan`, встроен в недельный `audit.yml`, так что
красный скан эскалируется в tracking-issue). Он сканирует **тот же
запиненный по digest сборочный образ**, в котором компилирует `build.py`:
digest читается из `packaging/Dockerfile` — единого источника правды, — так
что бамп базового образа автоматически перенацеливает скан. Находки Trivy
фильтруются до rpm-пакетов, реально поставляющих бандлённый `.so` (маппинг
soname→пакет выводится из самого образа через `rpm -qf` и держится в
синхроне с allow-list `.so` из `build.py` тестом-drift-guard), а
`--ignore-unfixed` оставляет только *фиксабельные* HIGH/CRITICAL CVE — так
что красный прогон есть actionable-сигнал к пересборке, а не шум. Reviewed
baseline `.trivyignore` фиксирует подтверждённые находки; после
восстановления 2026-06-28 (ниже) он **пуст**, так что любая фиксабельная
находка краснит гейт.

Триггер уже привёл к одному восстановлению: две HIGH-CVE в `openssl-libs`,
которые он вскрыл (`CVE-2026-45447`, затем `CVE-2024-4741`), закрыты
2026-06-28 бампом digest образа `manylinux_2_28` (`893d0c9d` → `e6cdd8b2`),
который несёт `openssl-libs 1:1.1.1k-16.el8_6` (фиксированный релиз).
Allow-list `.so` в `build.py` пере-засевать **не пришлось**: новый образ
сохраняет те же sonames (`libssl.so.1.1`/`libcrypto.so.1.1`, `libffi.so.6`,
…) и тот же flavor Python `cp313`, так что набор бандлённых `.so` не
изменился. Постоянным долгом владельца остаётся **внешнее действие
восстановления**, на которое указывает триггер при затронутом релизе:
сделать re-release (отозвать и переопубликовать затронутый артефакт) после
бампа digest. Замечание: бандлённый OpenSSL по-прежнему серии EOL `1.1.1k`,
поэтому будущие backport-CVE будут повторяться и сканер остаётся
несущим. Привязка скана к *dist-тарболу* вместо сборочного образа (точнее,
но у голых `.so` нет rpm-метаданных, через которые Trivy сопоставляет CVE) —
возможное будущее уточнение.

Одно известное расхождение image-vs-dist — `libmpdec`: скан сопоставляет его
с системным rpm `mpdecimal` (`/lib64/libmpdec.so.3`), но Nuitka вмораживает
*собранный из исходников* `libmpdec.so.4`, которым не владеет ни один rpm, —
так что скан отслеживает системную копию, а не точную отгружаемую. Направление
консервативное (у mpdecimal ничтожная CVE-поверхность) и это та же первопричина,
которую закрыло бы уточнение с dist-тарболом выше.

**Runtime-гард прав каталога dist (реализовано на `python-port`, ещё не
в релизе):**

В README к установке пользователям предписывается распаковывать бинарник в
каталог, принадлежащий root и недоступный для записи остальным. Это
предписание необходимо, но зависит от пользователя, поэтому
`packaging/entry.py` теперь подкрепляет его гардом времени загрузки
(`_guard_dist_dir_permissions`): когда frozen-бинарник работает от root, он
делает `stat` каталога dist (разрешённый родитель `sys.executable`, где
лежат соседние разделяемые объекты) и отказывается диспетчеризовать — с
действенной диагностикой, — если этот каталог не принадлежит root или
доступен на запись группе либо посторонним. Гард срабатывает только из
frozen-сборки (маркер Nuitka `__compiled__`) и только под root, так что
разработчик, запускающий дерево dist без привилегий, не затронут;
`FERM_SKIP_DIST_PERM_CHECK=1` отключает его для нестандартных раскладок. Он
живёт в `packaging/` (не в `src/pyferm/`), сохраняя инвариант Фазы 3. По
своей природе он остаётся сетью «best-effort»: ловит каталог dist, доступный
на запись не-root, но не разделяемый объект, уже подложенный туда до
проверки (разрыв time-of-check / time-of-use), и проверяет сам каталог dist,
а не его родителей — так что он дополняет, но не заменяет корректную
установку в каталог, принадлежащий root.

**Точный vs нормализованный allow-list `.so` (мелкий остаток):**

Allow-list нормализует минорную версию SONAME (`libssl.so.1.1` →
`libssl.so.1`), поэтому гипотетический `libssl.so.1.0` тоже прошёл бы гейт.
Сегодня это смягчено запиненным по digest сборочным образом (точный набор
`.so` фиксирован образом), поэтому записано как остаток, который стоит
ужесточить, если нормализация переживёт digest-пин, а не как открытая дыра.

### Фаза 4 — Операционная безопасность (движок diff/apply) ✅

Зависит от Фазы 2 (нативный nft: handles + атомарные транзакции).
Стратегия `commit`, вычисляющая и применяющая дельту вместо полного
flush-replace, плюс богатая история конфигов и резервные копии. Дешёвый
зачаток (`--backup-dir`) вынесен в Фазу 1.

Первый срез — **инкрементальное атомарное применение дельты** под `--nft` —
реализован в ветке `python-port` (ещё не выпущен): под `--nft` команда
`commit` по умолчанию выполняет инкрементальную nft-транзакцию (дельту,
вычисленную против живого снимка `nft list table`), а не `flush table` +
полная пересборка. Неизменные цепочки и неизменные именованные сеты не
трогаются, поэтому их порульные packet/byte-счётчики и kernel-состояние
выживают при перезагрузке ruleset. Флаг `--full-reload` возвращает полную
пересборку, применяемую как атомарная замена всей таблицы (`delete table` +
`add table`), а не простой `flush table`: `flush table` лишь очищает цепочки
от правил, сохраняя их объявления, поэтому base-цепочка, убранная из конфига,
выжила бы пустой-но-подключённой к hook (продолжая применять свою policy);
замена удаляет её по-настоящему. Следствие: элементы сета, добавленные в
таблицу ferm вне ferm (out-of-band), не переживают full-reload — их сохраняет
именно delta-путь. Первый запуск (снимка нет), пустой снимок
или дифф, содержащий retype/удаление типа/flags сета, детерминированно
уходит в full-reload; предикат fallback реализован единственной именованной
функцией. Дельта остаётся одной транзакцией `nft -f` (атомарность
сохраняется). Дельта конвергентна: элементы, присутствующие в ядре, но
отсутствующие в конфиге, удаляются — как при полной пересборке.

Второй срез — **история конфигурации и откат через etckeeper** — также
реализован в ветке `python-port` (ещё не выпущен). Вместо собственной VCS
поверх `--backup-dir` исходный конфиг версионируется etckeeper (стандарт для
`/etc`): каждое успешное применение фиксирует коммит в истории `/etc` с
семантическим сообщением о дельте ядра (переиспользуя движок `diff_tables` из
`--plan`), а `ferm rollback` возвращает `/etc/ferm` к прошлой ревизии
(только git, path-scoped) и переприменяет её. Коммит VCS-агностичен и
best-effort (сбой не трогает фаервол); `--no-etckeeper` отключает. Семантика
source-, не байт-точная: откат перегенерирует ruleset из откатанного
исходника. Детали для оператора — в README.

#### Отложенные пункты

Следующее явно вынесено за рамки Фазы 4 и записано здесь, чтобы не потеряться.

- *Порульная дельта по nft handle* — порульные packet/byte-счётчики
  выживают только в неизменных цепочках; изменённая цепочка сбрасывается и
  пересобирается целиком. Дифф отдельных правил по handle с сохранением их
  счётчиков — отдельная, более сложная работа (трекинг handle, порядок вставки).
- *История/резервные копии конфигов* — доставлено срезом etckeeper выше.
  Внутри него остаётся отложенным: числовой `ferm rollback -N` (на N ревизий
  назад; голая форма — ровно один шаг, глубже нужен `--to <sha>`);
  мультивендорный откат (адаптеры hg/bzr/darcs); узкий per-path коммит только
  `/etc/ferm` (etckeeper по конструкции коммитит весь `/etc`); и расширенный
  `--backup-dir` с кольцом снимков применённого ruleset (снято — откат через
  историю конфига перегенерирует ruleset, чего достаточно).
- *Сохранение внешнего dynamic-наполнения сетов* — сохранение элементов,
  добавленных внешним `nft add element` и неизвестных конфигу. Это
  конфликтует с конвергенцией (live перестаёт равняться desired), поэтому
  в базовый срез не входит; при необходимости — отдельный opt-in флаг позже.
- *append-only `--noflush`* под `--nft` — остаётся отложенным.
- *JSON-режим* (`nft -j`) — не относится к delta-пути.

### Фаза 5 — nft-нативная выразительность ✅

Зависит от Фазы 2 — ради этого и шли в нативный nft: сеты, словари
(maps), интервалы, конкатенации, нативный `reject-with` и прирост
производительности на роутерах/NAT-боксах.

Первый срез — **свёртка анонимных сетов** под `--nft` — реализован в
ветке `python-port` (ещё не выпущен): соседние листовые правила,
отличающиеся ровно одним set-пригодным значением, сворачиваются в одно
правило с анонимным сетом (`tcp dport { 22, 80, 443 }`), а
`ferm --plan --nft` честен относительно свёрнутой формы. Негированные
сопоставления и правила с поэлементно различными операторами остаются
линейными (safe-bias); бэкенд `iptables` и ядро ferm не тронуты.

Второй срез — **именованные сеты** (`@set` / `SetRef`) — тоже реализован в
ветке `python-port` (ещё не выпущен): определение `@set` в ferm эмитит
нативную декларацию `add set` / `add element`, ссылки транслируются в
операнд `@name`, а `ferm --plan --nft` разбирает блоки `add set` /
`add element` со стороны ядра, так что добавление сета, изменение набора
элементов и удаление каждое всплывает как `SetChange`. Имена и элементы
валидируются на границе эмиссии (fail-closed); бэкенд `iptables`
по-прежнему разворачивает значения инлайн на этапе разбора, поэтому
именованный сет там — no-op.

Третий срез — **интервальные сеты** под `--nft` — тоже реализован в ветке
`python-port` (ещё не выпущен): элемент сета, записанный как диапазон адресов
(`10.0.0.0-10.0.0.255`, диапазоны IPv6) или CIDR-префикс, помечает сет
`flags interval` наряду с числовыми диапазонами портов (`1024-2048`), которые
поддерживались и раньше. Поскольку ядро переписывает элементы при readback
(выровненный по префиксу диапазон схлопывается в CIDR, host-биты маскируются,
у `/32`-`/128`-хоста префикс отбрасывается), plan-дифф канонизирует обе
стороны к этой форме хранения, поэтому `ferm --plan --nft` по неизменному
интервальному сету сходится, а не показывает фантомный дифф. Перекрывающиеся
интервалы nft отвергает на этапе применения (`nft -c`), так что детекция
перекрытий остаётся задачей ядра (fail-closed).

Четвёртый срез — **нативный `reject-with`** под `--nft` — тоже реализован в
ветке `python-port` (ещё не выпущен): каждое значение `reject-with`, которое
принимают `iptables`/`ip6tables -j REJECT` (все канонические icmp/icmpv6-типы
и их короткие алиасы вроде `net-unreach`, `tcp-rst`, `no-route`),
транслируется в нативный nft `reject with icmp[v6] type ...` /
`reject with tcp reset`. nft пишет iptables-имя `icmp-proto-unreachable` как
`prot-unreachable`; все написания сверены с живым `nft -c`. Неизвестное
значение остаётся ошибкой трансляции (fail-closed), а не применяется молча.

Пятый срез — **verdict maps (vmap)** под `--nft` — тоже реализован в ветке
`python-port` (ещё не выпущен): пробег соседних одноключевых листовых правил,
различающихся И ключом, И verdict'ом, сворачивается в один verdict map
(`tcp dport vmap { 22 : accept, 80 : drop }`) — verdict-несущий аналог свёртки
анонимных сетов. Допустимы только чистые verdict'ы (`accept`/`drop`/`return`/
`jump`/`goto`); `reject`, `log` и NAT-операторы, которые nft внутри
vmap запрещает, разрывают пробег и остаются линейными, а дубль ключа
завершает пробег (nft отвергает vmap с дублями ключей). Члены упорядочены по
канонической ранжировке ключа (nft хранит vmap упорядоченным по ключу, как
сет), а IPv6-адресный ключ (несущий собственные двоеточия) разбивается по
разделителю членов ` : ` и канонизируется, поэтому `ferm --plan --nft` по
неизменному набору правил сходится. Многоматчевые правила и смесь свёрнутого
сета с одиночками сознательно отложены.

Чтобы `ferm --plan --nft` оставался честным, желаемый набор правил в реальном
прогоне пре-валидируется через `nft -c` перед диффом (пропускается под
`--test`, который использует фейковый nft): невыполнимый план — например,
цепочка `arp` с `tcp`-матчем, который nft отвергает как «conflicting
protocols», — прерывается с собственной диагностикой nft и кодом выхода 1,
вместо того чтобы быть показанным как actionable-изменение.

С анонимными, именованными и интервальными сетами, verdict maps и нативным
`reject-with` основная отдача нативной выразительности Фазы 5 по существу
завершена. Два пункта про maps/конкатенации из заголовка фазы — сознательные
задокументированные исключения, записанные ниже.

#### Отложенный долг

Следующие пункты явно **вне scope** Фазы 5 и записаны
здесь, чтобы не потеряться.

- *Форма port-range для iptables (`lo-hi` vs `lo:hi`)* — ferm эмитит для
  iptables форму `--dport lo-hi` (через дефис), которую современный
  `iptables-restore` на бэкенде `nft` отвергает (ему нужно `lo:hi`). Это
  унаследовано верно от Perl-оракула, поэтому изменение — осознанное
  решение о поведении, а не исправление бага: из-за него data-path
  port-range для iptables нетестируем на дистрибутивах с бэкендом `nft`.
  Трекать как решение о расхождении с оракулом.
- *Свёртка конкатенаций* — составной ключ-сет (`ip saddr . tcp dport
  { 1.2.3.4 . 22, ... }`) сворачивал бы соседние правила, ко-варьирующие сразу
  по двум селекторам («диагональ», которую картезианская свёртка сетов
  оставляет линейной). Семантически безопасно и выразимо в nft (проверено на
  живом `nft`), но отложено как низкоценная оптимизация: ferm-конфиги пишут
  списками правил по сервисам, поэтому диагональный паттерн возникает редко, а
  свёртка добавляет surface разбора/канона ради малой реальной выгоды.
  Вернуться, если конкретный конфиг это мотивирует.
- *Именованные словари (maps)* — неприменимо: в языке ferm нет конструкции
  словаря, поэтому бэкенду нечего эмитить как named map, а свою таблицу ferm
  владеет (flush-replace), так что readback ядра её никогда не содержит.
  Записано, чтобы пункт «maps» из заголовка Фазы 5 был явно учтён, а не потерян.

### Укрепление контрактов

Сквозной проход укрепления контрактов в ветке `python-port` (ещё не
выпущен): псевдоним типа `Family` и гейт `parse_family` заменили
разрозненные строковые проверки, обёртки-значения и rendered-объекты
(`Negated`/`Multi`/`SetRef`, `RenderedOption`/`RenderedRule`) были
заморожены ради value-based-равенства и неизменяемости связывания
(мутабельные скоупы `Option`/`Rule` намеренно не трогали), бэкенд
iptables получил валидаторы имён и политик, а утверждения на уровне
модуля документируют ключевые
инварианты-подмножества. Небольшой Hypothesis-suite покрывает границу
`Family` и контракты замороженных значений.

Валидатор имён iptables — намеренное fail-closed-отступление от
побайтового паритета: имя таблицы/цепочки должно соответствовать
безопасному алфавиту `[A-Za-z0-9_.+-]`, тогда как оракул проверяет только
длину и эмитит имя как есть. Это whitelist, а не blacklist, потому что одно
и то же имя попадает в два стока с разными опасными наборами — save-грамматику
`iptables-restore` на быстром пути (пробелы и управляющие байты разрывают
строки) и сырую подстановку в командную строку на медленном пути, которая
исполняется через `/bin/sh`. Blacklist, заточенный под save-грамматику,
пропускает в этот shell-сток любой shell-метасимвол (`;`, `$`, backtick,
`|`, ...), а правила `eb`/`arp` идут по медленному пути по умолчанию, так
что имя вроде `x;reboot` означало бы инъекцию команд от root. `+` остаётся
в алфавите, потому что оракул его принимает (например, `a+b`), сохраняя
паритет. Узкое переужесточение для имён, которых не бывает в реальных
конфигах (например, разделители save-грамматики `:`/`*`/`[`, которые
устанавливаются в ядро как часть имени без проблем), закреплено тестом
`test_ipt_name_rejects_grammar_separators_by_design`, чтобы позже его не
приняли за баг паритета.

Отложено (открытое решение по модели угроз): *значения* из конфига —
`saddr`, `comment`, log-префиксы, параметры match/target — проходят через
`shell_escape`, чей медленный режим (`_SLOW_SPECIAL_RE`) намеренно **не**
квотит `$`, backtick и `()`, побайтово верно перловому оракулу
(`system($str)` тоже использует `/bin/sh`). Значение без пробелов вроде
`$(id)` поэтому всё ещё доходит до шелла на медленном пути. Закрытие этого
означало бы квотирование этих метасимволов в `shell_escape` —
санкционированную дивергенцию от оракула, которая ломает golden-паритет и
требует перегенерации затронутых `.result`-файлов. Оставлено открытым до
решения о том, считает ли модель угроз ferm значения конфига (в отличие от
имён) недоверенным входом; апстрим-ferm — нет.

Тестирование конвергенции дельты рассматривалось как дополнительная
Hypothesis-цель (генерация пар `(snapshot, desired)` с проверкой, что
дельта совпадает со структурным реинтерпретатором). Оно не было добавлено:
`tests/unit/test_plan_delta_convergence.py` и opt-in сессия
`delta_apply_e2e` уже охватывают этот путь структурно и end-to-end;
генератор продублировал бы реинтерпретатор без нового сигнала.

Golden-файлы `--plan` и `--nft`
(`tests/golden/{plan,plan_nft,nft,delta_nft}/`) — это **детерминированные
самоснимки, не сверяемые с оракулом**: у Perl-оракула нет бэкендов
`--plan`/`--nft`, поэтому независимого эталона для этих слоёв не существует,
а зафиксированный ожидаемый вывод — собственный вывод pyferm, сверенный
вручную в момент написания. Они ловят *регрессию от* снимка (доказано:
реверс порядка элементов множества их роняет), но не снимок, *неверный с
рождения*. nft-слой частично заякорен opt-in предпроверкой `nft -c`,
per-PR-гейтом `delta_apply_e2e` (rootless readback живого `nft` на
data-path) и еженедельной сессией `nft_conformance` (дифференциал канона
против живого `nft`); слой iptables `--plan` заякорен слабее всех и
опирается на сверку при написании. Единственный якорь, всё ещё
отсутствующий в PR CI, — это per-PR-прогон `nft_conformance`, поэтому у
nft-*канона* (в отличие от data-path) есть лишь еженедельный
автоматический бэкстоп; его продвижение в PR CI — действующее средство, а
более тяжёлые docker-e2e/conformance-сьюты намеренно оставлены за
пределами PR CI.

### Фаза 9 — Операционные интеграции (хорошее поведение в системе)

Операционный клей вокруг stateless-CLI `ferm` — делает его лучшим
гражданином системы, в духе etckeeper-интеграции Фазы 4, а не расширения
DSL правил. Эти пункты в основном живут вне ядра (sidecar-скрипты,
пакетные артефакты, cron-задания); там, где нужна правка ядра, она мала и
отмечена явно.

**Prometheus-метрики через textfile-collector (sidecar; ядро остаётся
stateless).**

Наблюдаемость файрвола по паттерну node_exporter *textfile collector*:
cron-задание атомарно пишет файл `*.prom` в каталог коллектора (например,
`/var/lib/node_exporter/textfile_collector/`), а экспортер его подхватывает
— без HTTP-эндпоинта и без долгоживущего процесса ferm. Коллектор сэмплит
`nft -j list ruleset` (counters достаются бесплатно из nft-бэкенда) и отдаёт
агрегаты. Ядро остаётся stateless; единственное состояние — небольшой
локальный файл коллектора с предыдущим хешем рулсета, рядом с самим выводом
`.prom`.

Метрики отвечают на два вопроса — *когда* правила изменились и *как*
(грубо), — сознательно оставляя точный дифф существующей истории
конфигурации:

- **Когда — двое разных «часов».** Применение, инициированное конфигом,
  фиксирует своё время точно — его пишет сам apply-путь
  (`ferm_last_apply_timestamp_seconds`). Внешнее изменение (кто-то запустил
  `nft` руками) ловится коллектором сравнением хеша live-рулсета с
  предыдущим снимком, поэтому его время *приблизительно*, с точностью до
  интервала cron (`ferm_last_change_timestamp_seconds` плюс
  `ferm_ruleset_info{hash="…"}`, у которого метка перещёлкивается при
  изменении). Ценность — в связке: `last_change` сдвинулся, а `last_apply`
  нет — это сигнал внешнего дрейфа.
- **Как — только грубо.** Числа и направление, на том же движке
  `diff_tables`, что уже использует `--plan`: `ferm_drift` (live ≠ desired,
  0/1), `ferm_drift_rules{change="added|removed"}`,
  `ferm_rules_total{family,table,chain}`, `ferm_set_elements_total{set}`.
  Это показывает «+3 правила в INPUT, blocklist вырос на 200», но не *какие*
  правила — текст правила как метка был бы антипаттерном высокой
  кардинальности.
- **Точный текстовый дифф остаётся вне Prometheus.** Он уже живёт в истории
  etckeeper (Фаза 4): каждый apply коммитит семантическое сообщение,
  описывающее дельту ядра через тот же `diff_tables`. Разделение труда:
  Prometheus наводит на «когда + сколько + был ли дрейф?»; `git log`
  (etckeeper) / journald несёт точное «что».

Этот пункт делит свою механику — `diff_tables` и live-снимок nft — с
будущим режимом **drift-detection** (read-only `ferm --plan --check`,
отдающий контракт exit-кода для cron/monit: в синхроне / дрейф / ошибка),
поэтому оба — естественные соседи и стоят здесь рядом.

**Drift-detection как контракт exit-кода для cron/monit
(`ferm --plan --check`).**

Сегодня `ferm --plan` — это *диагностический* дифф для человека: печатает
расхождения live-vs-desired и возвращает 0 в норме (1 — только на
`@preserve`). Для cron/monit нужен *машинный* режим — exit-код как
контракт: `0 = в синхроне`, `2 = обнаружен дрейф`, `1 = ошибка` — ничего
не печатая (или одну строку сводки) вместо полного диффа. Это
единственная недостающая деталь; сам дифф уже считает `diff_tables`. ferm
остаётся stateless-CLI — без демона — а ритм задаёт `monit check program`
или строка cron. Это рантайм-двойник etckeeper: etckeeper ловит изменение
*файла конфига*, drift-detection — изменение *установленного состояния*.
Ядро задевается слегка (новый флаг и путь exit-кода поверх готового
диффа).

**stdin и shebang: существующий контракт для документирования, а не фича
для разработки.**

Оба уже работают механически и нуждаются лишь в задокументированном
контракте плюс e2e-тесте:

- *stdin* — `ferm -` читает конфиг из стандартного ввода и в Perl-оракуле,
  и в порте (`open_script`: `filename == '-'` → STDIN), так что
  `echo "…" | ferm -` работает уже сейчас; POD документирует это лишь
  вскользь.
- *shebang* — `ferm` берёт входной файл первым позиционным аргументом,
  поэтому скрипт с `#!/usr/bin/ferm` напрямую исполним. Единственная
  оговорка — классическое ограничение ядра на shebang: только один
  аргумент интерпретатора, поэтому `#!/usr/bin/ferm --nft` ненадёжен, а
  поддерживаемая форма — `#!/usr/bin/env -S ferm --nft`.

Работа здесь — задокументировать поддерживаемые паттерны и добавить
e2e-тест, что shebang-скрипт действительно применяет свои правила:
контракт, а не новое поведение. (Идея `--eval "строка"` отброшена: это
чистый сахар над уже работающим пайпом `ferm -` и лишь расширяла бы
поверхность CLI.)

**Dispatcher-хуки NetworkManager / systemd-networkd (пакетные
артефакты).**

Переприменение правил при up/down интерфейса — динамические имена
интерфейсов, VPN-туннели — через dispatcher-скрипты в
`/etc/NetworkManager/dispatcher.d/` и `networkd-dispatcher`, которые
вызывают `ferm` (опционально `--def DEV=$1`) на смене состояния. Это не то
же, что внутрицикловые `@hook pre/post/flush` ferm (shell *внутри* цикла
применения); dispatcher работает *снаружи*, по событию интерфейса, и затем
зовёт ferm. Правок ядра нет — пакетные хук-скрипты плюс пример. Реальная
боль, которую это закрывает: `@def $DEV`, зашитый в конфиг, ломается при
переименовании интерфейса.

**Рецепт сосуществования с CrowdSec / fail2ban (docs + пример).**

Современные ответчики на вторжения (nftables-bouncer CrowdSec, fail2ban)
пополняют nft named set, на который затем дропает правило. Фаза 5 дала
named sets, так что точка стыковки есть; сосуществование — через отдельные
nft-таблицы с per-table drop-terminal семантикой (ключевое слово `@extset`
рассмотрено и отклонено — таблица ferm, которая flush-replace'ит
собственное содержимое, не должна владеть set'ом, пополняемым извне). Это
задокументированный рецепт плюс пример (опционально `@hook`-sidecar в
стиле легаси-iptables-цепочки `DOCKER-USER`), а не код ядра.

**Валидация конфигов в pre-commit / CI (`--noexec`).**

Упаковать уже доступный `ferm --noexec --lines` (распарсить и отдать
рулсет, не трогая ядро) как хук `.pre-commit-hooks.yaml` и переиспользуемый
GitHub Action, чтобы `.ferm`-файлы валидировались в их собственных
репозиториях до деплоя. Тривиально — обёртка над существующим поведением,
расширяет аудиторию.

**Менее советую (записано, чтобы не возвращаться вслепую).**

- *Конфигурация ulogd2 для NFLOG/ULOG* — нишево; мало где используется.
- *Нативный Ansible-модуль* — полезно, но это отдельный проект на другом
  языке/в другой экосистеме, размывающий фокус; общую потребность в
  автоматизации закрывают dispatcher- и CLI-интеграции выше.

## Сквозная тема: IPv6

Поддержка IPv6 улучшается *поперёк* фаз, а не одним куском: Фаза 1
дословно сохраняет family-зависимое форматирование (оракул: `test/ipv6/`),
Фаза 5 даёт нативную выразительность, Фаза 6 добавляет анти-локаут-хелперы.
Отдельной «фазы IPv6» сознательно нет.
