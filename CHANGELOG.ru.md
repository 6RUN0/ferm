# История изменений

Все значимые изменения **порта ferm на Python** документируются в этом
файле.

Формат основан на [Keep a Changelog](https://keepachangelog.com/), а сам
проект придерживается [семантического версионирования](https://semver.org/).

Историю оригинальной реализации на Perl (версии вплоть до `v2.8`
включительно) см. в [`reference/NEWS`](reference/NEWS).

## [Не выпущено]

Порт на Python (`src/pyferm/`). Фаза 1 воспроизводит поведение
Perl-реализации и генерирует наборы правил `iptables`; её вывод проверяется
побайтово против Perl-оракула, хранящегося в `reference/`.

Фаза 2 добавляет **опциональный нативный бэкенд `nftables`** за флагом
`--nft`. Бэкенд по умолчанию остаётся `iptables`, поэтому существующие
конфигурации и вывод не меняются, пока не передан `--nft`.

### Добавлено — Фаза 2 (нативный nft-бэкенд)

- **Опциональный нативный nftables-бэкенд `--nft`.** Транслирует
  структурное правило в нативный набор правил nft и применяет его
  атомарно через `nft -f -`. Используется только *текстовый* протокол
  nft, без зависимости от сборки nft с JSON / `libjansson`. `--nft` —
  строго opt-in; по умолчанию остаётся бэкенд `iptables`.
- **`--nft` валидирует набор правил через `nft -c -f -` перед
  применением.** Applier сперва запускает текстовый `--check` nft
  (валидация против netlink, ничего не устанавливающая) и только при её
  успехе передаёт реальный `nft -f -`, выводя собственную диагностику nft
  *до* любого изменения ядра вместо общей ошибки применения.
- **`--nft --interactive --shell` выдаёт рабочую анти-локаут-сеть.**
  Сгенерированный shell-скрипт снимает снимок таблицы ferm
  (`nft list table`) перед применением, а после таймаута подтверждения
  удаляет только что применённую таблицу и перезагружает снимок
  (`nft -f`) — зеркало живого отката, так что администратор, не
  подтвердивший правила, восстанавливается. Скрипт также эхо-сообщает об
  откате в stderr, чтобы заглушённые (`2>/dev/null`) команды
  восстановления больше не откатывали администратора молча.

### Изменено — Фаза 2 (нативный nft-бэкенд)

- **Семантика `policy DROP` под `--nft` отличается.** nft-бэкенд владеет
  единственной таблицей `table <family> ferm` и не перехватывает
  монолитные ядерные цепочки `INPUT` / `FORWARD` / `OUTPUT` так, как это
  фактически делает плоский набор правил iptables. Поэтому базовые цепочки
  ferm сосуществуют с базовыми цепочками других таблиц на том же hook
  (упорядоченные по приоритету), и пакет может быть принят
  более приоритетной чужой цепочкой раньше, чем достигнет цепочки ferm.
  Следовательно, `policy DROP` в `table ip ferm` ведёт себя иначе, чем
  монолитный `INPUT DROP` в iptables. Это задокументированное, ожидаемое
  поведение модели собственной таблицы, а не баг; администраторам, которым
  нужна точная семантика монолитного DROP, следует оставаться на бэкенде
  `iptables` по умолчанию.

### Удалено / не поддерживается — Фаза 2 (нативный nft-бэкенд)

- **`@preserve` не поддерживается бэкендом `--nft`.** Использование
  `@preserve` вместе с `--nft` — чистая, явная ошибка, а не молчаливый
  no-op. Это намеренная регрессия opt-in-бэкенда; бэкенд `iptables` по
  умолчанию поддерживает `@preserve` ровно как прежде.

- **NAT с портом под `--nft` требует транспортного матча.** NAT-вердикт,
  отображающий порт (`DNAT to ...:port`, `SNAT to ...:port`,
  `REDIRECT`/`MASQUERADE to-ports`) без предшествующего матча
  `proto tcp`/`udp`, теперь даёт чистую ошибку ferm на этапе трансляции:
  nft отвергает такое отображение при применении (`transport protocol
  mapping is only valid after transport protocol match`) и иначе вызвал бы
  откат. Добавьте в правило матч протокола.

### Безопасность — Фаза 2 (нативный nft-бэкенд)

- **Операнды под `--nft` экранируются или валидируются до попадания в
  save-скрипт.** Раньше значение из конфига, содержащее пробел, `;`, `#`
  или двойную кавычку, могло вырваться из своего nft-токена — например,
  `saddr "1.2.3.4 accept;#" DROP` рендерился как
  `ip saddr 1.2.3.4 accept;# drop`, молча превращая правило `DROP` в
  `accept` (форма, которую `nft -c` принимает без нареканий). Имена
  интерфейсов теперь выдаются как экранированные nft-строки в кавычках
  (wildcard `*` сохраняется); адреса, порты, **протоколы**, лимиты
  скорости и идентификаторы цепочек проходят грамматическую проверку и
  дают чистую ошибку ferm вместо набора правил, который nft применил бы
  неверно. Операнд протокола (`proto "tcp accept;#" DROP` →
  `meta l4proto tcp accept;# drop`) был последним незащищённым стоком и
  теперь валидируется наравне с остальными. Бэкенд `iptables` по умолчанию
  уже экранировал эти операнды и никогда не был затронут.
- **Сбой снимка для отката больше не вырождается в разрушительное
  удаление.** Откат `--nft` удаляет собственную таблицу ferm, когда
  снимка нет (настоящий первый запуск); транзиентный сбой
  `nft list table` на *существующей* таблице теперь отличается от
  настоящего первого запуска (по ошибке, отличной от `ENOENT`) и
  прерывает выполнение до любого изменения ядра, вместо того чтобы быть
  принятым за «нет предыдущей таблицы» и удалить её при откате.

### Исправлено

- nft-бэкенд: рендер `reject-with tcp-reset` в семействе `ip6` (раньше был
  только для `ip`), как у дефолтного бэкенда и family-агностичного
  `reject with tcp reset` самого nftables.

### Добавлено — Фаза 1 (верный порт)

- **Front end языка конфигурации**, портированный с Perl: токенизатор и
  ленивый поток токенов, парсер рекурсивного спуска `enter()` по блокам,
  областям видимости и ключевым словам, copy-on-write область видимости
  переменных/функций/массивов и отложенная реализация значений
  (`@resolve()`, `@ipfilter()` и подобные раскрываются поздно).
- **Реестр определений модулей** и компактный DSL кодирования опций
  (эквиваленты `add_proto_def` / `add_match_def` / `add_target_def`), так
  что поддерживаемые модули netfilter переносятся из Perl-таблиц.
- **Сборка правил** — структура правила, развёртывание в декартово
  произведение списков опций, `format_option` и побайтово точный
  `shell_escape`.
- **Домены по семействам и неизменяемая модель `Options`**, со швом
  инъекции домены → бэкенд.
- **Бэкенд iptables** с обоими путями выполнения: `--fast` (сборка
  save-файла и его передача в `iptables-restore`, атомарно) и `--slow`
  (один вызов `iptables` на правило), а также генерация shell-скрипта
  через `--shell`.
- **CLI и верхнеуровневый поток**, включая `--noexec`, `--lines`,
  откат `--interactive` с таймаутом подтверждения и семейства
  `ip` / `ip6` / `arp` / `eb`.
- **`import-ferm`** — преобразует дамп `iptables-save` в конфигурацию ferm
  (круговое преобразование save → ferm → save).
- **Резолвер имён `@resolve`** на основе `dnspython`.

### Тестирование и инструментарий

- **Тестовая обвязка на golden-файлах**, проверяемая против Perl-оракула,
  с канонизацией недетерминированного порядка вывода таблиц/цепочек.
- **Дифференциальный фаззинг против Perl-оракула** (Hypothesis):
  токенизатор, `shell_escape`, лексика импорта, разбиение по обратным
  кавычкам, `@substr`, токены опций, ридер предыдущего состояния и целые
  сгенерированные грамматикой конфигурации — что привело к исправлениям
  обработки Unicode в `\s` (`re.ASCII`), Perl-нумификации, модели
  `substr`/undef и побайтово точных регулярных выражений разбора
  save-дампа.
- **Корпус реальных конфигураций**, компилируемых против оракула
  (fast + slow).
- **Crash-фаззинг через `atheris`** обоих парсеров и **контейнерный
  anti-lockout e2e** для `--interactive` (оба opt-in), а также
  периодическая сессия **мутационного тестирования `mutmut`**.
- **Контейнерный датапат-e2e** (`nox -s datapath_e2e`, opt-in): гоняет
  реальный трафик `nmap --reason` / `ncat` через установленные
  ferm-правила в топологии из трёх netns, проверяя поведение
  ACCEPT / DROP / REJECT / state / NAT и паритет бэкендов `--nft` и
  дефолтного. Расширяемая матрица дистрибутивов
  (`nox -s datapath_e2e_matrix`) прогоняет ту же обвязку на Debian
  (bookworm + trixie), Ubuntu, Fedora, Arch, Rocky и openSUSE Leap,
  определяя пакетный менеджер (apt / dnf / apk / pacman / zypper) — так
  что новый дистрибутив добавляется одной строкой.
- **Паритет диагностики**: golden-файлы фиксируют stderr для случаев
  negative / params / warning.
- **Побайтово точный ввод-вывод**: конфигурация, обратные кавычки,
  zonefile и `--def` (`argv`) читаются как байты latin-1 и переносятся
  через границы CLI, restore и `import-ferm`.
- **Ограничения `MAX_BLOCK_DEPTH`** на вложенность блоков парсера и
  **`MAX_VALUE_DEPTH`** на вложенность ридера значений — оба отказывают
  локализованной диагностикой вместо переполнения стека.

### Инфраструктура проекта

- Python-проект развёрнут в корне репозитория (`uv` + `src-layout`,
  `hatchling`) с объявленной поддержкой Python **3.11–3.14**.
- Оригинальная реализация на Perl перемещена в `reference/` в качестве
  семантического оракула.
- Гейты под управлением `nox` (lint, tests, typecheck, порог покрытия,
  matrix, fuzz, build, deps-lowest, проверка workflow), связанные в
  обязательный `preflight` и в CI GitHub Actions (статические проверки
  выделены отдельно, patch-гейт на PR, еженедельный audit + Dependabot).
